## 1. Диагноз (почему сейчас сломано)

Замеры showcase_book (A5, PNG 699×992 px, полоса набора по полям шаблона:
y = 118…803 px, т.е. top 2.5cm / bottom 4.0cm):

| Страница | До низа полосы | Внутренние дыры | Вердикт |
|---|---|---|---|
| p_2 (фронтиспис) | −3.2 мм (ОК) | **18 мм** воздуха вокруг иллюстрации | дыра |
| p_3 (текст) | −2.1 мм | 2.3 мм (межабзацный интервал) | идеал |
| p_4 (финал) | **−10.8 мм (~2–3 строки)** | 4.9 мм | недолив |

Плюс на каждой странице нижнее поле 40 мм — канон книжной вёрстки,
но на PNG-витрине читается как «текст висит в воздухе» (см. §8, вопрос 1).

**Главная поломка — архитектурная, и она общая для всех шаблонов.**
Текущий пайплайн: `server.py` пишет content.json → шаблон Typst льёт ВЕСЬ
текст одним потоком → Typst рвёт страницы где получится. Сервер не знает,
что такое страница: не знает её границ, ёмкости, не проверяет результат.
Любой шаблон с длинным текстом или картинками даёт ту же болезнь: недоливы,
дыры, блоки, перекинутые на следующую страницу «как повезло». Текущее «почти
хорошо» у книги достигнуто ручной подгонкой (убрали абзац, ужали картинку
88%→65%) — правкой голосом, что прямо нарушает смысл MCP.

## 2. Принцип (универсальный)

Страница — единица вёрстки в КАЖДОМ шаблоне. MCP обязан сам:

1. **ЗНАТЬ** границы и ёмкость страницы каждого шаблона/пресета до написания
   текста (Budget);
2. **ВЕРСТАТЬ** так, чтобы блоки не оставляли дыр (Compose);
3. **ПРОВЕРЯТЬ** результат каждой компиляции и отдавать числовой отчёт
   (Verify), а не полагаться на глаз человека или LLM.

Универсальный инвариант:

> Внутри непрерывного потока (между легальными разрывами) каждая страница
> заполнена до низа полосы набора. Частичная страница допустима только
> непосредственно ПЕРЕД легальным разрывом или в конце документа — и тогда
> к ней применяется политика финала шаблона.

Что считается легальным разрывом — решает не движок, а **паспорт страницы
шаблона** (§3.1): у книги это новая глава, у ГОСТа — каждый раздел с нового
листа, у письма разрывов нет вообще. Всё остальное — дефект вёрстки,
и сервер обязан его обнаружить и назвать числом.

## 3. Архитектура: универсальное ядро + паспорт шаблона

Ядро (калибровка, introspection, отчёт, серверная расстановка блоков) — одно,
живёт в `server.py` и ничего не знает про конкретные шаблоны. Все различия
шаблонов вынесены в декларацию.

### 3.1 Паспорт страницы шаблона (page contract)

Каждый шаблон объявляет в `data/styles.json` (рядом со своим пресетом) блок
`page_contract`:

```json
"page_contract": {
  "columns": 1,                  // 2 у resume (проверка по колонкам)
  "legal_breaks": ["chapter"],   // что даёт право на частичную страницу
  "exempt_zones": ["frontispiece", "chapter_end"],  // где воздух — дизайн
  "final_page_policy": "fill_or_ornament",  // или "partial_ok", "balance_columns"
  "min_final_fill": 0.6
}
```

Паспорта восьми шаблонов:

| Шаблон | Колонки | Легальные разрывы | Зоны-исключения (воздух = дизайн) | Политика финала |
|---|---|---|---|---|
| report | 1 | новый H1 (опционально) | титульная зона | fill ≥ 0.5 или блок выводов |
| academic_ru | 1 | **каждый H1-раздел — с нового листа (ГОСТ 7.32 п.6.2.2)**; конец раздела частичный — легален | титульный лист | по ГОСТу, частичный конец раздела — норма |
| book | 1 | новая глава | фронтиспис/шмуцтитул | добить текстом или оформить концовку |
| technical | 1 | новый H1 (опционально) | титул | fill ≥ 0.5 |
| portfolio | 1 + сетки галерей | новый проект/кейс | hero-зона | галерея масштабируется под остаток |
| letter | 1 | нет | блок подписи/реквизитов | частичная страница — НОРМА (исключение) |
| journal | meander-полосы | новая статья | заголовочный разворот | полоса до низа (meander уже обеспечивает) |
| resume | 2 | нет (одна страница) | шапка | **баланс колонок**: разница низа колонок ≤ 3 строк |

Верификатор (§3.4) читает паспорт и применяет общие правила — никакой
шаблонной логики в ядре.

### 3.2 Budget — инструмент `estimate_page_budget`

```
estimate_page_budget(template: str, preset_overrides: dict = {}) → {
  lines_per_page: int,       # строк на полной текстовой странице
  words_per_page: int,       # слов (эмпирически, для языка документа)
  chars_per_line: float,
  columns: int,              # из паспорта
  image_cost: spec           # (width_pct, aspect) → строк вытеснения
}
```

Работает для ЛЮБОГО шаблона одинаково — **эмпирическая калибровка**, не
расчёт по метрикам шрифта (метрики хрупки: переносы, justify, costs):

1. Сервер компилирует одноразовый калибровочный документ (длинный lorem
   языка документа, без картинок) данным шаблоном/пресетом.
2. Через introspection (§3.4а) снимает факт: строк на полной странице,
   средняя длина строки, слов на странице (для resume — на колонку).
3. Кэш в `data/calibration.json`, ключ = (template, sha256(preset), lang).
   Инвалидация по mtime/sha шаблона.

Вытеснение картинкой: `iw = ширина_полосы × pct; ih = iw / aspect;`
`строк = ceil((ih + подпись + 2×отбивка) / leading)`.

Использование (зашить флоу в SKILL.md): автор запрашивает бюджет ДО написания
текста и пишет В РАЗМЕР: `target_words = pages × words_per_page − Σ image_cost`.

### 3.3 Compose — блоки не оставляют дыр

Болезнь общая: шаблоны вставляют `figure()`/галереи/таблицы в поток; если блок
не влез в остаток страницы, Typst переносит его целиком и оставляет дыру.

Решение: **позицию `auto` любого невлезающего блока вычисляет СЕРВЕР,
а не Typst** — для всех шаблонов единообразно:

- Шаблон умеет только исполнять точную позицию `after:N` (в book уже есть;
  довести до остальных, включая таблицы и галереи).
- Сервер по данным introspection первого паса знает, какой абзац на какой
  странице и сколько места осталось → решает: блок встаёт здесь (влезает
  с запасом ≥ 2 строки текста после) или первым на следующей странице,
  а текст подтягивается вперёд и доливает текущую.
- Сложное обтекание — meander (journal уже на нём).

Сопутствующие мелкие правки: нормировать отбивки вокруг top-блоков
фиксированными интервалами (~0.8em), убрать растягивающиеся `v(..., weak)` —
источник дыры 18 мм на фронтисписе книги.

### 3.4 Verify — introspection + отчёт заполненности (каждая компиляция)

**а) Метки в Typst.** ВСЕ шаблоны ставят после каждого абзаца
`#metadata((i: idx)) <bp-para>`. После компиляции сервер вызывает
`typst query main.typ "<bp-para>"` → для каждой метки (page, y).
Карта: какой абзац на какой странице, где его нижняя граница. Точно, из PDF.

**б) PIL-контроль (второй рубеж).** Перенести чернильный замер из
render_all.py в server (`_measure_fill`): по PNG на страницу — последняя
строка с чернилами, максимальная внутренняя дыра. Для resume — отдельно
по каждой колонке (маска по x-диапазонам колонок из паспорта). Ловит то,
что метки не видят (картинки, орнаменты, таблицы).

**в) Отчёт в ответе `compile_preview` / `compile_pdf`** — новое поле,
для любого шаблона:

```json
"layout_report": {
  "pages": [
    {"page": 2, "fill": 0.97, "lines_short": 1,
     "holes_mm": [{"after_para": 0, "mm": 18}], "verdict": "HOLE"},
    {"page": 3, "fill": 0.99, "verdict": "OK"},
    {"page": 4, "final": true, "fill": 0.84,
     "words_to_add": 60, "verdict": "FINAL_PARTIAL"}
  ]
}
```

Правила verdict (генерик, параметры из паспорта):
- страница внутри непрерывного потока: последняя базовая линия ≥
  (низ полосы − 1.5×leading) → OK; иначе **UNDERFILLED** + `lines_short`
  + `words_to_add`;
- внутренняя дыра > 2.5×leading вне `exempt_zones` → **HOLE** + где и сколько;
- страница перед легальным разрывом / финальная: по `final_page_policy`
  и `min_final_fill` → OK или **FINAL_PARTIAL** + рекомендация;
- resume: |низ левой − низ правой| > 3 строк → **UNBALANCED_COLUMNS**.

**г) Жёсткий режим.** `compile_pdf(..., strict_layout=true)`: при любом
verdict ≠ OK — ошибка с полным отчётом. Для витрин и CI.

### 3.5 Двухпасовая компиляция (как 3.3 и 3.4 склеиваются)

```
пас 1: компиляция с auto-позициями → typst query → карта абзацев/остатков
       → сервер расставляет блокам точные after:N
пас 2: компиляция с точными позициями → layout_report → ответ инструмента
```

Пасы — внутри одного вызова compile_*, наружу виден только отчёт.

### 3.6 Авто-подгонка (v2, опционально, после ручного флоу)

Если после паса 2 verdict ≠ OK, сервер пробует ограниченный перебор
в пределах эстетики: ширина блока ±15 п.п. (диапазон 55–95%), сдвиг точки
вставки ±2 абзаца. До ~6 комбинаций. Не сошлось — честный отчёт с точным
дефицитом в словах. Текст сервер НЕ сочиняет никогда: добавка слов — работа
автора (LLM) по числам из отчёта.

## 8. Решения Натали (2026-06-11)

1. **Нижнее поле книги — канон 40 мм остаётся.** Но именно поэтому критично,
   чтобы ВСЕ страницы доходили текстом до низа полосы набора одинаково:
   при большом поле любой недолив (2–3 строки) виден как «разное расстояние
   снизу» и ломает единство разворотов. Требование: разброс позиции последней
   строки между страницами одного потока ≤ 1 строки.
2. **Финал — добивать до низа полосы**, чтобы все страницы смотрелись едино,
   а не оформлять орнаментом. (Орнамент остаётся как опция паспорта для
   других случаев.)

Практическое следствие, проверенное на сказке: недолив последней страницы
лечится не только словами — рычаг и размер иллюстрации (80% vs 65% леса дал
ровно 2 недостающие строки). Авто-подгонка §3.6 должна крутить оба рычага.

## Принцип «хвоста» / Overflow philosophy

Журнальный и книжный материал **льётся со страницы на страницу** — это нормально и правильно.

Задача MCP — сделать **каждую страницу красивой как отдельный блок**:
- Страница наполнена полностью (≥ 95 % fill) и скомпонована аккуратно.
- Текст, который не вошёл, переходит на следующую страницу — и там тоже компонуется как блок.
- Заголовок новой главы **не обязателен** на каждой странице. Хвост статьи продолжается без заголовка — ровно так, как в реальном журнале или книге.

QC проверяет каждую страницу независимо. Страница с хвостом (24 %, 30 % fill) — это **ещё не скомпонованная страница**, а не ошибка. Следующий шаг — добрать её до полного блока.
